Anforderungen
4. "Fehler-Suche: Der korrupte Analyst"
   - Ziel: Finde alle 5 Rechenfehler in einer fertigen NWA innerhalb von 90 Sekunden.
   - Spielmechanik: 
     - Eine komplette Tabelle wird gezeigt.
     - Manche Werte stimmen nicht (z.B. Gewichtung 30% * Note 10 = 300 statt 3).
     - Spieler klickt auf Zellen.
     - Messbarkeit: Gefundener Fehler = +20%. Falscher Klick (korrekte Zelle) = -10%. Ziel: 100%.
   - Lernziel: Kritische Prüfung von Ergebnissen, Genauigkeit.

Implementierungsplan (UX-Global-Optimiert)
1. Informationsarchitektur und Fokus-Hierarchie
- [ ] Fokus: Eine "gefälschte" Matrix (Rechenfehler oder Logikfehler).
- [ ] Werkzeug: Roter Korrekturstift.
- [ ] Szenario: "Ihr Kollege hat die Zahlen geschönt. Finden Sie den Fehler."

2. Screen-Layout
- [ ] Dokumenten-View (Excel-Lookalike).
- [ ] Toolbar: Stift, Radierer.
- [ ] Fehler-Counter (z.B. "Noch 3 Fehler versteckt").

3. Browser- und Device-Resilienz
- [ ] Pinch-to-Zoom für Details.
- [ ] Zellen müssen groß genug sein, um sie anzutippen.

4. Kernvisuals
- [ ] "Kritzel"-Optik für die Markierungen (SVG Paths).
- [ ] Stempel "REJECTED" wenn Fehler gefunden.

5. Interaktion und Feedback
- [ ] One-Tap Interaction: Tippe auf die falsche Zelle -> Wird eingekreist.
- [ ] Feedback: "Richtig! 3 * 4 ist nicht 15."
- [ ] Bei Falsch-Klick: "Nein, diese Berechnung stimmt."

6. KPI-Klarheit
- [ ] Zeitdruck (optional): "Meeting beginnt in 60s".
- [ ] Score basierend auf Geschwindigkeit und Genauigkeit.

7. QA-Paket
- [ ] Finger-Wisch vs Tippen unterscheiden.
- [ ] Lesbarkeit der "kleinen Zahlen" prüfen.
