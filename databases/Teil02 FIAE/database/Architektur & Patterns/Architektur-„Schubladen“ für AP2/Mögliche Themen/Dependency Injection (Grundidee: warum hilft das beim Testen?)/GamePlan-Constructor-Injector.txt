Anforderungen (reverse engineered)
- Ziel: Beherrschung des Inversion of Control (IoC) Prinzips durch Dependency Injection (DI) zur Entkopplung von Systemkomponenten.
- Kontext: Software-Architektur, Testbarkeit (Mocking), Wartbarkeit.
- UX-Prinzip:
  - "The Universal Socket": Eine Maschine (Klasse) hat kein fest verbautes Stromkabel mehr. Sie besitzt eine universelle Buchse (Interface).
  - "Module-Swap": Der Spieler kann verschiedene Stecker (Abhängigkeiten) einstecken: Den "Real-Power-Plug" (Datenbank), den "Battery-Dummy" (Mock) oder das "Solar-Panel" (Cloud-Service).
  - "Testing-Light": Sobald der Dummy eingesteckt wird, leuchtet eine "Test-Bereit" Lampe auf.

Arbeitspakete (UX-Global-Optimiert)
1. Informationsarchitektur und Fokus-Hierarchie
- [ ] Sichtprioritaeten: 1 = Die universelle Buchse (Parameter), 2 = Die auswechselbaren Stecker (Objekte), 3 = Die interne Logik (Maschine).
- [ ] Textbudget: Max. 10 Codezeilen für den Konstruktor-Vergleich.
- [ ] "Decoupling-Visual": Visuelle Lücke zwischen Maschine und Stromquelle, die nur durch Injektion geschlossen wird.

2. Screen-Layout (Container-Ebenen reduzieren)
- [ ] Struktur: `page -> machine-blueprint -> injection-dock -> component-inventory`.
- [ ] Blueprint: Transparente Ansicht der Klasse mit den offenen Steckplätzen (Desktop mittig, Mobile 50dvh Höhe).
- [ ] Inventory: Auswahl der verfügbaren Instanzen am unteren Rand (Drag & Drop fähig).

3. Browser- und Device-Resilienz
- [ ] `min-height: 100dvh` Standard.
- [ ] Mobile: "Tap-to-Snap" Mechanik zum Verbinden von Komponenten.
- [ ] `safe-area-inset` für das Inventar-Menü.

4. Kernvisuals
- [ ] Elektronik-Ästhetik: Platinen-Layouts, Steckverbindungen, leuchtende Kontaktpunkte (shared_theme.css).
- [ ] "Connection-Spark": Kurzes Aufblitzen beim erfolgreichen Einstecken einer Abhängigkeit.
- [ ] High-Contrast Icons für Interface-Typen (z. B. `<I> DBService`).

5. Interaktion und Feedback
- [ ] Match-Feedback: "Klick! Das `IShippingService` Interface akzeptiert den `DHLProvider`. Die Maschine läuft."
- [ ] Test-Mode: Animation, wie ein Mock-Objekt (Dummy-Stecker) die Datenbank-Abfrage simuliert, ohne das echte System zu belasten.
- [ ] Erfolg: "Klasse entkoppelt! Sie kann jetzt mit jedem Stecker betrieben werden, der in die Buchse passt."

6. KPI-Klarheit
- [ ] Degree-of-Decoupling: "Entkopplungs-Grad: 100%".
- [ ] Testability-Boost: "Automatisierbarkeit: Maximal".

7. Content-/Copy-Feinschliff
- [ ] Reale Fälle: `Logger`, `DatabaseService`, `PaymentProvider`, `EmailClient`.
- [ ] Tonalität: Technisch, modular, flexibel.

8. QA-Paket
- [ ] Screenshot-Check: Sind die Interface-Beschriftungen auf einem kleinen Handy-Bildschirm erkennbar?
- [ ] Test: Bleibt die Maschine "stehen", wenn die Abhängigkeit wieder herausgezogen wird (Null-Safety-Warnung)?

9. Abnahme-Kriterien
- [ ] Der Spieler versteht DI als physisches "Baukasten-Prinzip" für Software.
- [ ] Das Interface wirkt wie ein exklusives Engineering-Tool für System-Architekten.
- [ ] Klarheit über den massiven Vorteil für das Unit-Testing.
