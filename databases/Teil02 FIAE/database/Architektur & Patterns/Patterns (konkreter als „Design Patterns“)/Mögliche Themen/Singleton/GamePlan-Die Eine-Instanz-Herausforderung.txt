Anforderungen (reverse engineered)
- Ziel: Garantie, dass von einer Klasse systemweit nur eine einzige Instanz existiert, um Ressourcen zu bündeln oder globalen Zustand zu kontrollieren.
- Kontext: Ressourcen-Management, Konfiguration, Logging, Hardware-Zugriff.
- UX-Prinzip:
  - "The Highlander": Es kann nur eine Instanz geben.
  - "Protected Gate": Der Versuch, ein neues Objekt mit `new` zu bauen, prallt an einer geschlossenen Pforte (private Constructor) ab.
  - "Universal Key" (`getInstance`): Jeder bekommt denselben Schlüssel zu demselben, bereits existierenden Raum.

Arbeitspakete (UX-Global-Optimiert)
1. Informationsarchitektur und Fokus-Hierarchie
- [ ] Sichtprioritaeten: 1 = Die einzige Instanz (The Unique One), 2 = Die blockierte `new`-Operation, 3 = Der statische Zugriffspunkt.
- [ ] Textbudget: Max. 5 Code-Fragmente für die Singleton-Implementierung.
- [ ] "Identity-Visual": Anzeige der Objekt-ID (Hashcode), die bei jedem Zugriff identisch bleibt.

2. Screen-Layout (Container-Ebenen reduzieren)
- [ ] Struktur: `page -> world-access -> singleton-throne -> blocker-gate`.
- [ ] Throne-Area: Ein prominenter Platz in der Mitte, der von der Instanz besetzt ist (Desktop 50% Breite, Mobile 60dvh Höhe).
- [ ] Access-Log: Zeigt die IP-Adressen oder Thread-IDs, die alle zum selben Thron pilgern.

3. Browser- und Device-Resilienz
- [ ] `min-height: 100dvh` Standard.
- [ ] Mobile: "Tap-to-Access" Mechanik – Teste von verschiedenen Stellen den Zugriff.
- [ ] `user-select: none` für die zentralen UI-Komponenten.

4. Kernvisuals
- [ ] Epische Ästhetik: Marmor, Gold, Lichtkegel oder futuristische Energie-Singularität (shared_theme.css).
- [ ] Abwehr-Animation: Blitze oder Schilde zucken auf, wenn jemand versucht, ein zweites Objekt zu erzeugen.
- [ ] High-Contrast IDs für den Identitätsnachweis.

5. Interaktion und Feedback
- [ ] Access-Feedback: "Bestätigt. Du hast die Instanz #001 erhalten. Genau wie alle anderen vor dir."
- [ ] Block-Feedback: "Halt! Du kannst keinen weiteren `ConfigManager` erschaffen. Der Thron ist besetzt."
- [ ] Erfolg: "Ressourcen geschont! Alle Teile der App nutzen denselben Speicher für die Log-Einträge."

6. KPI-Klarheit
- [ ] Instance-Count: "Instanzen im System: 1".
- [ ] Resource-Efficiency: "Speichernutzung: Minimal".

7. Content-/Copy-Feinschliff
- [ ] Reale Fälle: "DatabaseConnectionPool", "ApplicationSettings", "PrintSpooler".
- [ ] Tonalität: Erhaben, kontrolliert, einzigartig.

8. QA-Paket
- [ ] Screenshot-Check: Ist der "Abwehr-Blitz" bei einer `new` Operation auf dem Handy visuell befriedigend?
- [ ] Test: Erscheint die Warnung vor "Global State" (Nachteile von Singleton) zum richtigen Zeitpunkt?

9. Abnahme-Kriterien
- [ ] Der Spieler versteht das Singleton als mächtiges, aber vorsichtig einzusetzendes Werkzeug.
- [ ] Das Interface wirkt wie ein exklusiver Tempel der Objekt-Kontrolle.
- [ ] Höchste Klarheit über die `private static` Logik.
