Anforderungen (reverse engineered)
- Ziel: Sicherstellung der Schnittstellen-Compliance durch Validierung von JSON-Payloads gegen formale Schemata (OpenAPI/JSONSchema).
- Kontext: API-Labor, Fokus auf Datentypen und Struktur.
- UX-Prinzip:
  - "The Contract" (Schema) ist das statische Gesetz (linke Spalte).
  - "The Incoming Data" (Payloads) fließen in der rechten Spalte ein.
  - "Scanner-Interaktion": Der Spieler aktiviert einen Scan, der Fehler im JSON-Code rot markiert.

Arbeitspakete (UX-Global-Optimiert)
1. Informationsarchitektur und Fokus-Hierarchie
- [ ] Sichtprioritaeten: 1 = Payload-Fehler (rotes Highlight), 2 = Schema-Regel (Referenz), 3 = Validitäts-Status.
- [ ] Textbudget: Nur essentielles JSON-Snippet zeigen, kein "Deep Nesting" Ballast.
- [ ] Keine Intro-Container; der erste "Broken" Payload wartet bereits auf den Scan.

2. Screen-Layout (Container-Ebenen reduzieren)
- [ ] Struktur: `page -> api-inspector -> side-by-side-diff -> status-badge`.
- [ ] Inspector-Area: Desktop 50/50 Split, Mobile vertikale Kartenfolge mit "Compare"-Modus.
- [ ] Controls: "Accept" / "Reject" Buttons großflächig am unteren Rand.

3. Browser- und Device-Resilienz
- [ ] `min-height: 100dvh` Standard.
- [ ] Code-Font: Verwendung von Fira Code o. ä. für maximale Lesbarkeit von Sonderzeichen.
- [ ] Mobile: "Mini-Diff" Ansicht (nur geänderte/falsche Zeilen einblenden).

4. Kernvisuals
- [ ] Scanner-Linie: Eine horizontale Lichtlinie fährt über den Code bei der Validierung.
- [ ] Error-Bubbles: Text-Popups direkt an der Zeile ("Expected 'integer', got 'string'").
- [ ] High-Contrast Code-Theming ( shared_theme.css).

5. Interaktion und Feedback
- [ ] Sofort-Scan: Reaktion innerhalb von <200ms nach Tastendruck.
- [ ] Feedback: "Stopp! Das Feld 'email' fehlt im Payload, ist aber im Vertrag als 'required' markiert."
- [ ] Lerneffekt: Schema-Snippet leuchtet bei Fehlersuche parallel zur fehlerhaften Zeile auf.

6. KPI-Klarheit
- [ ] Schema-Compilance: "X / Y Payloads korrekt beurteilt".
- [ ] Bug-Detection-Speed: "Mittlere Reaktionszeit: 1.2s".

7. Content-/Copy-Feinschliff
- [ ] Realistische API-Szenarien: "User-Registration", "Payment-Callback", "IoT-Sensor-Data".
- [ ] Tonalität: Technisch, präzise, analytisch.

8. QA-Paket
- [ ] Screenshot-Check: Ist der Code auch auf kleinen Displays ohne extremes horizontales Scrollen lesbar (Line-Wrap)?
- [ ] Test: Funktionieren die klickbaren Fehlermarkierungen auch auf Touch?

9. Abnahmekriterien
- [ ] Das Verständnis für strickte Schnittstellenverträge wird durch die "Scanner-Metapher" vertieft.
- [ ] Das Interface wirkt wie ein professionelles API-Debugging-Tool (ähnlich wie Postman/Insomnia).
- [ ] Reibungslose, hochperformante Code-Analyse.
