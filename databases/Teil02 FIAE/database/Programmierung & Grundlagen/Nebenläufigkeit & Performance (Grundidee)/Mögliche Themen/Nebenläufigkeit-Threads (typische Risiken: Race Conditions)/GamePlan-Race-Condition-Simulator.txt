Anforderungen (reverse engineered)
- Ziel: Praktische Demonstration und Behebung von Nebenläufigkeitsfehlern (Race Conditions) in Multithreading-Szenarien.
- Kontext: Parallelprogrammierung, Thread-Sicherheit, Ressourcen-Management.
- UX-Prinzip:
  - "The Two-Way Switch": Zwei Rivalen (Threads) versuchen gleichzeitig, denselben Lichtschalter (Variable) zu drücken. Ohne Koordination bleibt das Licht aus oder flackert undefiniert.
  - "The Shield" (Lock): Der Spieler kann einen Schutzschild (Mutex) um den Schalter aktivieren. Nur wer den Schild hält, darf schalten.
  - "State-Corruption Visual": Bei einem Konflikt fängt der Wert an zu "rauschen" (Glitch-Effekt) und verliert an Stabilität.

Arbeitspakete (UX-Global-Optimiert)
1. Informationsarchitektur und Fokus-Hierarchie
- [ ] Sichtprioritaeten: 1 = Die umkämpfte Variable (Spielfeld), 2 = Die Pfade der Threads (Akteure), 3 = Der Synchronisations-Status (Locked/Unlocked).
- [ ] Textbudget: Max. 10 Codezeilen für die Thread-Logik.
- [ ] "Critical Section" Highlight: Der gefährdete Bereich im Code leuchtet rot pulsierend.

2. Screen-Layout (Container-Ebenen reduzieren)
- [ ] Struktur: `page -> thread-lanes -> shared-resource-center -> sync-toolbox`.
- [ ] Lanes: Zwei parallele Pfade (Desktop links/rechts, Mobile oben/unten).
- [ ] Resource-Center: Das zentrale Element (die Variable), auf dem beide Pfade zusammentreffen.

3. Browser- und Device-Resilienz
- [ ] `min-height: 100dvh` Standard.
- [ ] Mobile: "Tap-to-Lock" Mechanik – Schnelles Reagieren per Daumendruck.
- [ ] `will-change: transform` für flüssige Thread-Animationen.

4. Kernvisuals
- [ ] Energie-Ästhetik: Zwei Lichtstrahlen (Threads) fließen auf ein Ziel zu.
- [ ] "Glitch-Pulse": Bei einer Race Condition verzerrt sich das UI kurzzeitig (digitale Störung).
- [ ] High-Contrast Symbole: Ein Vorhängeschloss für `Synced` und ein Warnungs-Blitz für `Insecure`.

5. Interaktion und Feedback
- [ ] Live-Clash: Die Threads "knallen" ungebremst aufeinander, wenn kein Lock gesetzt wurde -> Datenwert wird ungültig.
- [ ] Erfolg: "Sicherer Zugriff! Der Lock hat Thread B warten lassen, bis Thread A fertig war."
- [ ] Deadlock-Warnung: "Vorsicht! Wenn zwei Threads sich gegenseitig aussperren, friert das System ein."

6. KPI-Klarheit
- [ ] Thread-Safety: "Stabilitäts-Index: 100%".
- [ ] Collision-Counter: "Anzahl verhinderter Datenfehler: X".

7. Content-/Copy-Feinschliff
- [ ] Szenarien: "Bank-Kontostand-Update", "Ticket-Counter", "Logistik-Bestand".
- [ ] Tonalität: Intensiv, technisch, risikobewusst.

8. QA-Paket
- [ ] Screenshot-Check: Ist der "Glitch-Effekt" auf dem Handy-Display als Absicht erkennbar (kein Bug)?
- [ ] Test: Funktioniert die Thread-Verzögerung (Wait) bei aktivem Lock flüssig?

9. Abnahmekriterien
- [ ] Der Spieler erlebt die Gefahr paralleler Zugriffe als physische Kollision.
- [ ] Das Interface wirkt wie ein futuristischer Simulator für Quanten- oder Cloud-Systeme.
- [ ] Absolut klares Verständnis für die Notwendigkeit von `Synchronized` / `Atomic` Operations.
