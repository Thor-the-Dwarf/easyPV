Anforderungen (reverse engineered)
- Ziel: Frühzeitige Fehlererkennung und Durchsetzung von Coding-Standards durch statische Code-Analyse.
- Kontext: Linting (ESLint, Pylint), Clean Code, Automated Testing.
- UX-Prinzip:
  - "The Grammatic Guard": Der Linter ist der strenge Lektor, der über die "Grammatik" des Codes wacht, noch bevor er "ausgesprochen" (ausgeführt) wird.
  - "Squiggly-Hunt": Der Spieler jagt die berühmten roten/gelben Wellenlinien im Code.
  - "Rule-Matching": Jede Warnung muss der richtigen Regel (z. B. `no-unused-vars`, `eqeqeq`) zugeordnet werden.

Arbeitspakete (UX-Global-Optimiert)
1. Informationsarchitektur und Fokus-Hierarchie
- [ ] Sichtprioritaeten: 1 = Die Wellenlinie (Der Verstoß), 2 = Die Regel-Vorschläge, 3 = Der Auto-Fix Button.
- [ ] Textbudget: Max. 8 Zeilen Code-Snippet pro Level.
- [ ] "Rule-Library": Kurze 1-Satz-Erklärungen, warum eine Regel wichtig ist.

2. Screen-Layout (Container-Ebenen reduzieren)
- [ ] Struktur: `page -> ide-window -> lint-report-drawer -> fix-palette`.
- [ ] IDE-Window: Dunkler Hintergrund mit Syntax-Highlighting (Desktop 70% Breite, Mobile Full-Width).
- [ ] Palette: Auswahl der korrektiven Maßnahmen am unteren Rand (Buttons).

3. Browser- und Device-Resilienz
- [ ] `min-height: 100dvh` Standard.
- [ ] Mobile: "Tap-to-Fix" Geste direkt auf der Wellenlinie.
- [ ] `font-family: 'JetBrains Mono' or similar` für authentisches IT-Feeling.

4. Kernvisuals
- [ ] IDE-Ästhetik: Zeilennummern, Tabs, Dateibaum-Icons (shared_theme.css).
- [ ] "The Squigglies": Animierte rote Wellen unter dem fehlerhaften Code.
- [ ] Lint-Badge: Ein kleiner Counter oben rechts "Warnings: X / Errors: Y".

5. Interaktion and Feedback
- [ ] Fix-Feedback: "Klick! `==` wurde durch `===` ersetzt. Typ-Sicherheit wiederhergestellt!"
- [ ] Erfolg: "Linting bestanden. Dein Code ist jetzt sauber, konsistent und sicher."
- [ ] Rule-Info: Tooltip-Anzeige der offiziellen Linter-Dokumentation beim Hovern.

6. KPI-Klarheit
- [ ] Code-Cleanliness: "Qualitäts-Status: 100% Clean".
- [ ] Rule-Knowledge: "X / Y Linter-Regeln gemeistert".

7. Content-/Copy-Feinschliff
- [ ] Szenarien: "Ungenutzte Imports", "Fehlende Semikolons", "Potenzielle Null-Pointer", "Naming-Convention Brüche".
- [ ] Tonalität: Präzise, lehrreich, ordnungsliebend.

8. QA-Paket
- [ ] Screenshot-Check: Sind die Wellenlinien auch auf einem kleinen Smartphone-Display gut sichtbar und tappbar?
- [ ] Test: Stimmt das Syntax-Highlighting für verschiedene Programmiersprachen (JS/Python)?

9. Abnahme-Kriterien
- [ ] Der Spieler versteht Linting als hilfreichen Freund, nicht als lästiges Hindernis.
- [ ] Das Interface wirkt wie eine professionelle Entwicklungsumgebung.
- [ ] Höchste Klarheit über den Unterschied zwischen Warnung und Fehler.
