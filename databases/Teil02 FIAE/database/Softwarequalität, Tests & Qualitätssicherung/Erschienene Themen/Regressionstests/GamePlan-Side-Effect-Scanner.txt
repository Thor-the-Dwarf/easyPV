Anforderungen (reverse engineered)
- Ziel: Identifikation und Behebung von Seiteneffekten (Regressionen) nach Code-Änderungen durch gezielte Wiederholungstests.
- Kontext: CI/CD, Regressions-Testing, Impact Analysis, Testautomatisierung.
- UX-Prinzip:
  - "The Regression Radar": Eine Änderung ist ein Einschlag im System. Der Radar scannt die gesamte Oberfläche nach "Echos" (neuen Fehlern) in entfernten Modulen.
  - "Impact-Heatmap": Module, die eng mit der Änderung verknüpft sind, glühen heißer auf und erfordern dringendere Tests.
  - "Pipeline-Run": Der Spieler triggert die Test-Batterien und beobachtet die "grüne Welle" (Erfolg) oder den "roten Alarm" (Regression).

Arbeitspakete (UX-Global-Optimiert)
1. Informationsarchitektur und Fokus-Hierarchie
- [ ] Sichtprioritaeten: 1 = Die Code-Änderung (Zentrum), 2 = Die abhängigen Module (Satelliten), 3 = Der Pipeline-Status.
- [ ] Textbudget: Max. 3 Worte pro Modul-Name (z.B. "Payment Gateway", "User Auth").
- [ ] "Impact-Score": Prozentuale Wahrscheinlichkeit, dass Modul X durch Änderung Y bricht.

2. Screen-Layout (Container-Ebenen reduzieren)
- [ ] Struktur: `page -> dependency-radar-view -> test-selection-panel -> deployment-status`.
- [ ] Radar-View: Eine kreisförmige Map der Modul-Architektur (Desktop 60% Breite, Mobile Full-Screen).
- [ ] Panel: Checkliste der verfügbaren Automatisierungs-Scripts am Rand.

3. Browser- und Device-Resilienz
- [ ] `min-height: 100dvh` Standard.
- [ ] Mobile: "Pinch-to-Zoom" in der Modul-Map; Tap-to-Trigger für Tests.
- [ ] CSS-Animationen für den rotierenden Radar-Strahl.

4. Kernvisuals
- [ ] Cyber-Security Ästhetik: Dunkler Hintergrund, grüne Scan-Linien, blinkende Warnpunkte (shared_theme.css).
- [ ] "Ripple-Effect": Animierte Kreise, die von der geänderten Stelle ausgehen.
- [ ] High-Contrast Status-Icons: Kreuz (Fail), Haken (Pass), Kreis (Running).

5. Interaktion und Feedback
- [ ] Scan-Feedback: "Ping! Die Änderung am 'Login-Service' hat ein Echo im 'Warenkorb-Modul' ausgelöst. Regressionstest wird empfohlen."
- [ ] Pipeline-Moment: "Simulation läuft... Test 14/20 fehlgeschlagen! Regression im Checkout gefunden. Fix erforderlich."
- [ ] Erfolg: "Radar sauber! Alle Seiteneffekte wurden durch die Regression-Suite abgefangen. Sicher für Deployment."

6. KPI-Klarheit
- [ ] Regression-Coverage: "X % des Impact-Radius abgedeckt".
- [ ] Build-Stability: "Pipeline-Status: STABLE".

7. Content-/Copy-Feinschliff
- [ ] Szenarien: "Datenbank-Migration", "API-Update", "Refactoring der Utility-Klasse", "Security-Patch".
- [ ] Tonalität: Wachsam, automatisiert, prozesssicher.

8. QA-Paket
- [ ] Screenshot-Check: Sind die Verbindungen im Dependency-Radar auf einem Smartphone noch entwirrbar?
- [ ] Test: Sind die Auswirkungen der Test-Selektion (Zeitersparnis vs. Risiko) pädagogisch spürbar?

9. Abnahme-Kriterien
- [ ] Der Spieler versteht, dass jede Änderung ein Risiko für das Unbeteiligte birgt.
- [ ] Das Interface wirkt wie eine hochmoderne DevOps-Zentrale.
- [ ] Hohes Bewusstsein für den Wert von automatisierten Tests in der Wartungsphase.
